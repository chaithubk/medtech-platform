## 1. Opportunity

In a multi-repo, multi-service MedTech program, **integration failures are the most expensive defects**. A contract schema regression in `medtech-telemetry-contract`, a model payload format change in `medtech-edge-analytics`, or a broken MQTT dependency chain can each silently corrupt the end-to-end clinical workflow. In a medical device program, the cost of discovering such failures in production — rather than in CI — is not just engineering rework; it is potential harm to patients depending on the alarm system.

The **MedTech Platform** is the full-system integration and simulation environment. It uses Docker Compose to orchestrate the complete MedTech service stack — vitals publisher, edge analytics, clinician UI — in a reproducible, CI-executable environment that mirrors the production device runtime. It serves as:

1. The **gated integration layer** that must pass before any service release is promoted
2. The **simulation anchor** that developers use to run and validate the full stack locally without Yocto hardware
3. The **program health dashboard** that shows whether all services are contract-aligned and interoperating correctly

Without this platform, each service team would test only in isolation. The platform is where the cross-repo contracts, latency SLAs, and alarm chain integrity are validated end-to-end.

### Business Value

| Value | Description |
|---|---|
| Shift-left integration testing | Integration defects caught in PR CI, not production deployment |
| Hardware-independent full-stack validation | Developers validate the complete system without Yocto or physical devices |
| Release confidence gating | Service image promotion requires platform CI passage |
| Contract drift early warning | Platform-level manifest detects contract version misalignment across all service submodules |

## 5. Scope

### In Scope

- Docker Compose orchestration (`docker-compose.yml`, `docker-compose.build.yml`)
- Pinned GHCR image tags for all services (managed by CD pipeline)
- Integration test suite (`tests/integration_test.py`) validating:
  - MQTT vitals publication
  - MQTT prediction publication
  - All required contract fields present and typed correctly
  - End-to-end latency within SLA
- CI workflow (`ci.yml`) with:
  - Contract version parity check across service submodules
  - Integration test execution
- CD workflow (`cd-sync-images.yml`) for release-aligned image pin updates
- Platform-level contract manifest (`contracts/platform-contract-manifest.json`)
- Release workflow for platform image publishing

### Out of Scope

- Yocto/QEMU device runtime (see `medtech-device-os`)
- Service application logic (each service is a pinned image)
- Cloud backend orchestration (see `medtech-telemetry-cloud`)
- Multi-patient concurrent simulation

## 6. Functional Requirements

### FR-001: Full-Stack Compose Orchestration

`docker-compose.yml` MUST define three services (`vitals-publisher`, `edge-analytics`, `clinician-ui`) with:
- Pinned GHCR image tags
- Correct MQTT topic environment variables
- `depends_on: service_healthy` dependency chain
- A shared `medtech-network` bridge network

### FR-002: Integration Test Suite

`tests/integration_test.py` MUST validate:
1. MQTT broker reachability
2. Vitals payload receipt on `medtech/vitals/latest`
3. Presence of all required vitals fields: `timestamp`, `hr`, `bp_sys`, `bp_dia`, `o2_sat`, `temperature`, `quality`, `source`
4. Prediction payload receipt on `medtech/predictions/sepsis`
5. Presence of all required prediction fields: `timestamp`, `risk_score`, `risk_level`, `confidence`, `model_latency_ms`
6. End-to-end latency from vitals publish to prediction publish < 500 ms

### FR-003: Contract Parity CI Check

`ci.yml` MUST include a `contract-check` job that:
- Reads `contracts/platform-contract-manifest.json`
- Reads each service submodule's contract pin file
- Fails if any submodule's pinned contract tag differs from the manifest's `expected_contract_tag`
- Writes a parity table to `$GITHUB_STEP_SUMMARY`

### FR-004: CD Image Sync

`cd-sync-images.yml` MUST:
- Trigger on service release events
- Resolve the release-tagged commit SHA for each service
- Update the pinned image tag in `docker-compose.yml`
- Open a PR with a deterministic branch name (idempotent — no duplicate PRs)

### FR-005: Platform Contract Manifest

`contracts/platform-contract-manifest.json` MUST include:
- `expected_contract_tag`
- Service submodule pin file locations
- `last_updated` timestamp

## 8. Regulatory & Standards Alignment

| Standard | Relevance to This Product |
|---|---|
| **IEC 62304:2015 §5.6 / §5.7** | Integration testing is a formal IEC 62304 activity. The platform's integration test suite constitutes the software integration test evidence. Automated CI execution makes this evidence reproducible and traceable. |
| **ISO 14971:2019 §10** | Integration test results are verification records for risk controls (e.g., "end-to-end latency < 500 ms" verifies the risk control for delayed alarm delivery). |
| **IEC 60601-1-8:2006+AMD1:2012** | The 500 ms end-to-end latency target is directly derived from alarm system response time requirements. Integration tests enforce this as a gated CI requirement. |
| **HL7 FHIR** | Validated MQTT payload field alignment (via contract parity check) ensures that downstream FHIR mapping in the cloud backend is not broken by field renames or type changes. |

## 9. Risks & Mitigations (ISO 14971 Format)

| Risk | Likelihood | Severity | Risk Control |
|---|---|---|---|
| Service image tag updated without passing integration tests | Low | High (broken production stack) | CD PR requires CI passage before merge |
| Contract drift across service submodules undetected | Medium | High (parse failures in production) | `contract-check` CI job with mandatory pass gate |
| Integration tests check only happy path | Medium | Medium (latent defects in edge cases) | Test matrix includes `sepsis_onset` and `critical` scenarios |
| `cd-sync-images.yml` creates duplicate PRs | Low | Low (noise) | Idempotent branch naming + open PR check |
| `docker compose up` broken on developer workstation | Medium | Medium (lost developer productivity) | `README.md` quick-start tested in CI smoke job |
